
쿠버네티스 이벤트(Kubernetes Events)**는 클러스터 내에서 무언가 중요한 일(상태 변화, 에러, 라이프사이클 단계 등)이 발생했을 때 이를 기록하고 보고하는 특별한 종류의 리소스 객체입니다.

쉽게 말해 쿠버네티스 클러스터의 시스템 로그 또는 알림 창구 같은 역할을 하며, 관리자가 클러스터나 애플리케이션에 무슨 일이 일어나고 있는지 파악할 때 가장 먼저 확인하는 핵심 디버깅 도구입니다.

이벤트의 핵심적인 특징과 사용법을 정리해 드리겠습니다.

1. 이벤트의 주요 특징
상태 비저장(Stateless) 알림: 이벤트 자체는 클러스터의 '원하는 상태(Desired State)'를 정의하지 않습니다. 단지 과거에 일어났던 사실을 기록할 뿐입니다.
짧은 수명 (TTL): 이벤트가 계속 쌓이면 내장 데이터베이스인 etcd의 용량이 가득 차버리기 때문에, 기본적으로 생성된 지 1시간이 지나면 자동으로 삭제됩니다. 장기 보관이 필요하다면 별도의 로깅 시스템(Elasticsearch, Fluentd 등)이나 이벤트 익스포터(Event Exporter)로 외부로 빼내야 합니다.
컴포넌트들이 직접 발생시킴: kube-scheduler(스케줄링 성공/실패 기록), kubelet(컨테이너 이미지 풀링, 시작, 재시작, OOM 등 기록), kube-controller-manager 등 클러스터를 구성하는 각 컴포넌트가 맡은 역할에 맞게 직접 이벤트를 생성합니다.
2. 이벤트의 구조 (주요 속성)
이벤트를 조회해 보면 다음과 같은 정보가 포함되어 있습니다.

Type (유형): 크게 두 가지로 나뉩니다.
Normal: 정상적인 동작 과정 (예: 스케줄링됨, 이미지 다운로드 완료, 컨테이너 시작됨)
Warning: 주의가 필요한 예외 상황이나 에러 (예: 이미지 다운로드 실패, 메모리 부족(OOM), 노드 연결 끊김, 볼륨 마운트 실패)
Reason (이유): 짧은 단어로 된 이벤트의 원인 코드로, 스크립트나 자동화 도구로 파싱하기 좋습니다. (예: Scheduled, Failed, BackOff, Pulled)
Message (메시지): 사람(운영자)이 읽을 수 있는 상세한 설명입니다.
Source (출처): 이 이벤트를 누가 보고했는지 나타냅니다. (예: kubelet, default-scheduler)
Object (대상 객체): 이 이벤트가 어떤 리소스(Pod, Node, Deployment 등)와 관련된 것인지 가리킵니다.
